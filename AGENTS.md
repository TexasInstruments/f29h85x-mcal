# AGENTS.md

This file provides guidance to AI assistants helping users of the TI F29x MCAL product.

## Product Overview

The F29x MCAL (Microcontroller Abstraction Layer) is an AUTOSAR 4.3.1 compliant software package for TI F29H85x, F29P32x, and F29P58x devices. Package structure: `drivers/` (MCAL implementations), `plugins/` (EB Tresos configuration), `examples/` (bare-metal applications), `docs/User_Guide/` (documentation).

**Documentation for AI Agents:**

- `docs/User_Guide/llms.txt` - Concise summary for quick lookups and high-level overviews
- `docs/User_Guide/llms-full.txt` - Comprehensive documentation for detailed module descriptions, configuration parameters, and complete API references

## AI Agent Rules

### Configuration File Restrictions

> ## ⚠️ WARNING: NEVER directly edit these files:

1. **`.xdm` files** - EB Tresos configuration source files (require GUI modification)
2. **Auto-generated files** - Generated by EB Tresos from `.xdm` configurations:
   - Named: `<ModuleName>_Cfg.h`, `<ModuleName>_Cfg.c`, `<ModuleName>_Lcfg.c`, `<ModuleName>_PBcfg.c`
   - Located in: `*_Config/output/include/` and `*_Config/output/src/`
   - Manual changes will be overwritten on regeneration

**When configuration changes are needed, provide step-by-step EB Tresos GUI instructions:**

Example format:

``` text
To enable the SPI interrupt, make the following changes in EB Tresos:
1. Open EB Tresos and load your project
2. Navigate to the Spi module in the project explorer
3. Expand SpiGeneral container
4. Locate the SpiInterruptMode parameter
5. Change the value from current setting to LEVEL
6. Click Settings → AutoCalc to calculate derived values
7. Save the configuration (Ctrl+S)
8. Right-click project → Generate Project

Reference: Parameter definitions in plugins\Spi_TI_F29H85x\config\Spi.xdm
```

**Guidelines for providing instructions:**

1. Summarize the required changes
2. Specify module, container path, and parameter names
3. Indicate exact values to be set
4. Include AutoCalc and Generate steps
5. Reference parameter definitions: `plugins\{ModuleName}_TI_F29H85x\config\[Cdd_]{ModuleName}.xdm`

**When suggesting EB Tresos settings:**

1. Always suggest modules in order: Resource Allocator → Mcu → Port → Other peripherals
2. Check the module's XDM file at `plugins\{ModuleName}_TI_F29H85x\config\[Cdd_]{ModuleName}.xdm` before suggesting parameter values
3. **Skip parameters that have `<a:a name="VISIBLE" value="false"/>`** - these are hidden/internal parameters not shown in the EB Tresos GUI
4. When suggesting settings in a container, suggest them in the sequence of available parameters
5. Do not skip any **visible** parameters in the container - provide values for all visible parameters in order
6. For **reference-type parameters** (e.g. `DioChannelId`, `PortPinRef`), provide the full reference path to the target container that must be configured (e.g. `DioConf_DioChannel_LED_PIN`) so the user knows which object to create/select

### Mandatory Practices

- **Use F29H85x as target** in EB Tresos for all device variants (F29P32x/F29P58x included)
- **Run AutoCalc before Generate** in EB Tresos (Settings → AutoCalc)
- **Use McalLib for delays** - Call functions from [`drivers/Mcal_Lib/src/Mcal_Lib.c`](drivers/Mcal_Lib/src/Mcal_Lib.c):
  - `McalLib_DelayMsec(delayMsec)` - Millisecond delays
  - `McalLib_DelayUsec(delayUsec)` - Microsecond delays
  - `McalLib_Delay(loopcnt)` - CPU cycle delays (4 cycles per loop)
- **Use AppUtils for debug output** - Call `AppUtils_Printf()` from `examples/AppUtils/` instead of custom UART implementations
- **Default UART settings** (unless user specifies otherwise):
  - Baud Rate: 115200 (NEVER use 9600)
  - Data Bits: 8, Parity: None, Stop Bits: 1
- **Never use SysCfg** - Not compatible with MCAL; use EB Tresos exclusively
- **Never use BSW_Stubs in production** - `drivers/BSW_Stubs/` is for bare-metal examples only; use BSW vendor implementations in AUTOSAR projects
- **Only use modules from drivers/** - Do not introduce external AUTOSAR BSW modules or third-party implementations

### Workflow Guidelines

- **Scan examples directory first** - Find similar examples in `examples/` before implementing code to maintain consistency
- **Verify driver init requirement** - Check for `<ModuleName>_Init()` in `drivers/<ModuleName>/src/<ModuleName>.c` before configuring EcuM initialization
- **Driver initialization order** - Always: Resource Allocator → Mcu → Port → Other peripherals
- **Always ask for pin selection** - Pin assignments depend on hardware design; always ask users which pins to use
  - **F29H85X-SOM-EVM suggested pins (confirm with user before using):**
    - User LED3 (Green): GPIO23 (active low)
    - User LED4 (Green): GPIO9 (active low)
  - For other boards: Request pin mapping from user or schematic

### Technical References

- **Symbolic names for APIs** - Follow AUTOSAR TPS_ECUC_02108:
  - Format: `{Mip}Conf_{ContainerDef}_{ContainerValue}`
  - Example: `DemConf_DemEventParameter_CORTSI_E_CORE_FAILURE_1`
  - Ensure EB Tresos container shortName matches API call names
- **Device properties** - Determine from ResourceAllocator:
  - Read Device, Variant, Package from `config/ResourceAllocator.xdm`
  - Use corresponding properties files under `plugins/ResourceAllocator_TI_F29H85x/resources/`
  - Path pattern: `plugins/ResourceAllocator_TI_F29H85x/resources/{Device}/{Variant}/ResourceAllocator_{Package}.properties`
  - Example: For `Device=F29H859TU_Q1`, `Variant=F29H859TU8_Q`, `Package=F29H859TU8_Q_256ZEX`:
    - `plugins/ResourceAllocator_TI_F29H85x/resources/F29H859TU_Q1/ResourceAllocator_F29H859TU_Q1.properties`
    - `plugins/ResourceAllocator_TI_F29H85x/resources/F29H859TU_Q1/F29H859TU8_Q/ResourceAllocator_F29H859TU8_Q.properties`
    - `plugins/ResourceAllocator_TI_F29H85x/resources/F29H859TU_Q1/F29H859TU8_Q/ResourceAllocator_F29H859TU8_Q_256ZEX.properties`
  - Pin mux modes: `drivers/Port/include/Port_Pin_Map.h`
- **Adding modules to CCS projects** - Modify `.project` file in CCS project directory if build fails due to missing modules
- **Crash detection** - Program counter at these ROM addresses indicates crash (bootrom wait boot mode):
  - `0x0001535C` (PG1.0/1.1) or `0x0001549C` (PG2.0) - Wait boot mode, erased flash, or Link0 boot failure
  - `0x00014EB0` (PG1.0/1.1) or `0x00015040` (PG2.0) - Link1 boot failure

## Module Reference

### MCAL Modules (AUTOSAR Standard)

| Module | Description | When to Use |
| --- | --- | --- |
| **Can** | CAN 2.0 A/B, ISO 11898-1:2015, CAN FD (64 bytes, 5 Mbps). Interrupt/polling mode. | Vehicle networks, diagnostics, inter-ECU messaging |
| **Dio** | Digital I/O abstraction for channels, ports, channel groups | LEDs, switches, buttons, digital sensors, GPIO operations |
| **Fls** | Internal data flash read/write/erase/protect. Asynchronous operations. | Non-volatile storage, calibration data, configuration parameters |
| **Gpt** | General Purpose Timers (CPU-Timer0/1/2). One-shot and continuous modes. | Time-based scheduling, periodic tasks, timeout monitoring |
| **Lin** | LIN 1.3/2.0/2.1 protocol. Up to 20 kbps. | Low-cost, low-speed communication (seat controls, mirrors, sensors) |
| **Mcu** | MCU initialization: clock, PLL, RAM, reset, power modes | Foundation module - initialize first before any other driver |
| **Port** | PORT structure initialization. Pin multiplexing and configuration. | Configure pins for GPIO, CAN, SPI, ADC, LIN. Initialize after Mcu, before peripherals |
| **Spi** | Channel-based SPI access. 1-16 bit words, 16-level FIFO. | High-speed communication with EEPROMs, sensors, displays |
| **Wdg** | Watchdog timer. Reset mode, interrupt mode, windowed operation. | System safety monitoring, software hang detection |

### CDD Modules (Complex Device Drivers - TI Specific)

| Module | Description | When to Use |
| --- | --- | --- |
| **Cdd_Adc** | 12/16-bit ADC. 80 channels, 32 SOCs, oversampling, PPB, safety checkers. | Analog sensors (temperature, pressure, voltage), motor control feedback |
| **Cdd_Dma** | Direct Memory Access. High-speed data transfers without CPU intervention. | Memory-to-memory, peripheral-to-memory, memory-to-peripheral transfers |
| **Cdd_Ecap** | Enhanced Capture. Period/duty cycle measurement, edge detection, timestamping. | Speed measurement, pulse width/period measurement, position sensing |
| **Cdd_I2c** | I2C controller mode. 7/10-bit addressing. | I2C sensors, EEPROMs, RTCs |
| **Cdd_Ipc** | Inter-Processor Communication. 12 bytes/message, 255 channels. | Multi-core applications, AUTOSAR-SDK data exchange |
| **Cdd_Pwm** | Enhanced PWM. Simple mode (basic) and advanced mode (trip zones, dead-band). | Motor control, power conversion, LED dimming |
| **Cdd_Sent** | SENT protocol. Standard and MTP modes, 4 sensors/instance. | SENT-compatible sensors (position, pressure, temperature) |
| **Cdd_Uart** | UART up to 25 Mbps. 16-deep FIFOs, 5-8 bits, parity, stop bits. | Debug console, external device communication |
| **Cdd_Xbar** | Signal routing crossbar. Input, Output, CLB, ePWM, MINDB, ICL crossbars. | Flexible signal routing, GPIO-peripheral connections |

### Utility Modules

| Module | Description | When to Use |
| --- | --- | --- |
| **Mcal_Lib** | Common utility functions including `McalLib_Delay()` | Use for timing delays instead of NOP instructions |

> **Note:** All modules support AUTOSAR 4.3.1. Vendor ID: 44 (Texas Instruments).

## Workflows

### Setup

**Prerequisites:**

1. TI CGT C29 Clang Compiler (`c29clang`) - version in release notes
2. Code Composer Studio (CCS) - for debugging
3. EB Tresos Studio - for configuration (request from [MCAL-EB-TRESOS](https://www.ti.com/drr/opn/PROCESSOR-SDK-MCAL-EB-TRESOS))
4. CMake - for command-line builds

**Installation:**

1. Download MCAL package from [ti.com](https://www.ti.com/tool/F29-SDK)
2. Run installer
3. Copy `{MCAL_INSTALL_PATH}/plugins/*` to `{EB_INSTALLATION_PATH}/tresos/plugins/`

**License Setup:**

1. Install EB Client License Administrator and activate with provided key
2. Install EB Tresos Studio (version from release notes)

### Configuration

**Create New EB Tresos Project:**

1. Launch EB Tresos → File → New → Configuration Project
2. Select **F29H85x** as Target (for all device variants)
3. Add required modules (Resource Allocator, Mcu, Port, peripheral drivers)
4. Select "Default Recommended Configuration" when available

**Configure and Generate:**

1. Configure parameters in module GUI
2. Click Settings → AutoCalc to calculate derived values
3. Right-click project → Verify Project
4. Right-click project → Generate Project

Generated files in `output/`:

- `output/include/*_Cfg.h` - Configuration headers
- `output/src/*_PBcfg.c` - Post-build configuration
- `output/swcd/*_BSWMD.arxml` - BSW Module Description

**Load Example Configuration:**

1. Launch EB Tresos → File → Import
2. Browse to example configuration folder (e.g., `examples/Can/Can_Example_Classic_FD/F29H85x/Can_Example_Classic_FD_Config`)
3. Select configuration project to import

### Build

**CMake:**

Configure:

```shell
cmake -S . --preset mingw-c2000-f29h85x -DCMAKE_BUILD_TYPE=Debug -DBUILD_CONFIGURATION=FLASH -DTCP="C:\ti\ti-cgt-c29_<version>"
```

Build:

```shell
cmake --build build/artefacts -t <ExampleName>
```

**CMake Options:**

| Option | Values | Description |
| --- | --- | --- |
| `TCP` | Path | Compiler path (required if not in system PATH) |
| `CMAKE_BUILD_TYPE` | Debug, Release | Debug: symbols; Release: optimization |
| `BUILD_CONFIGURATION` | RAM, FLASH | RAM: debugging; FLASH: production (requires signing) |

**Code Composer Studio:**

1. File → Import → CCS Projects
2. Browse to `examples/<Module>/<Example>/CCS/`
3. Select `.projectspec` file
4. Build and debug from CCS

### Debug

**Troubleshooting:**

| Problem | Solution |
| --- | --- |
| **EB Tresos:** Plugins not appearing | Verify plugins copied to `{EB_INSTALLATION_PATH}/tresos/plugins/` |
| **EB Tresos:** Validation errors | Run Settings → AutoCalc before generating |
| **Build:** Compiler not found | Set `-DTCP=<path>` or add compiler to system PATH |
| **Build:** Version mismatch errors | Regenerate configuration files |
| **Build:** Signing fails (FLASH) | Check Python is installed |
| **Runtime:** Driver init fails | Verify Mcu_Init() called first; check clock configuration |
| **Runtime:** Peripheral not responding | Verify Port configuration for peripheral pins |
| **Runtime:** Program enters wait mode | Emulation API detected unavailable resource; check device variant |

**Debugging Tips:**

1. Enable DET: Set `<MODULE>_DEV_ERROR_DETECT = STD_ON` for runtime checks
2. Use AppUtils: `AppUtils_Printf()` outputs to UART
3. Check linker map: Verify section placement in `build/assets/<example>.map`

### Integration

**AUTOSAR Integration with Third-Party BSW:**

TI MCAL modules can be imported into other AUTOSAR tools (Vector DaVinci, ETAS ISOLAR-AB):

**Required files:**

1. MCAL Export (ARXML) - Export from EB Tresos via Im- and Exporters
2. Vendor-Specific Module Definition - `plugins/<Module>_TI_F29H85x/config/<Module>.arxml`
3. BSWMD File - Generate via Build Project → Generate_SWC-T

> **Note:** MCAL configuration must be done in EB Tresos. Import into other tools is for reference only.

**Device Variant Emulation:**

For F29P32x/F29P58x variants, call Emulation API before ECU initialization:

```c
DeviceSupport_EmulateDevice(DEVICE_F29P329SM2);
```

See `docs\User_Guide\_sources\src\Getting_Started.md.txt` for available device macros.

**Important Integration Notes:**

- MCAL runs on CPU1 only (CPU2/CPU3 not supported in current release)
- Linker files are for superset variants - modify for specific device memory sizes
- DeviceSupport code is reference only - replace with custom startup code for production

## Quick Reference

### Driver Initialization Pattern

```c
// 1. Initialize MCU (clocks, reset)
Mcu_Init(&Mcu_Config);
Mcu_InitClock(McuClockSettingConfig_0);
while (Mcu_GetPllStatus() != MCU_PLL_LOCKED) {}
Mcu_DistributePllClock();

// 2. Initialize Port (GPIO configuration)
Port_Init(&Port_Config);

// 3. Initialize peripheral drivers
Spi_Init(&Spi_Config);
```

### AUTOSAR API Pattern

All MCAL drivers follow consistent naming:

- `<Module>_Init(ConfigPtr)` - Initialize driver
- `<Module>_DeInit()` - De-initialize driver
- `<Module>_GetVersionInfo(versioninfo)` - Get version
- `<Module>_GetStatus()` - Get current status

### File Naming Conventions

| File Pattern | Description |
| --- | --- |
| `*.xdm` | EB Tresos configuration source |
| `*_Cfg.h` | Generated compile-time configuration |
| `*_PBcfg.c` | Generated post-build configuration |
| `*_BSWMD.arxml` | BSW Module Description |
